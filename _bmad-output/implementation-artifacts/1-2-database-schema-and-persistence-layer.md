# Story 1.2: Database Schema and Persistence Layer

Status: ready-for-dev

## Story

As a developer,
I want the short_urls table created and a repository layer for CRUD operations,
So that short URL mappings can be reliably stored and retrieved.

## Acceptance Criteria

1. **Given** the Fastify server starts with a valid DATABASE_PATH **When** the application initializes **Then** the SQLite database and short_urls table are created via Drizzle ORM migration
2. **Given** the database is initialized **Then** the `short_urls` table has columns: `id` (integer primary key), `short_code` (text, unique, indexed), `original_url` (text, not null), `normalized_url` (text, not null, unique), `created_at` (timestamp, not null)
3. **Given** the repository layer **Then** it exposes methods to insert a new short URL record, find by short_code, and find by normalized_url
4. **Given** the repository methods **Then** they return typed domain records (`ShortUrlRecord`), not raw SQL driver objects
5. **Given** the repository and database **When** unit/integration tests are run **Then** tests verify repository insert and lookup operations against a temp SQLite database

## Dev Notes

### Architecture Compliance

- **ORM**: Drizzle ORM `0.45.1` with `drizzle-kit` `0.31.10` вЂ” already installed as dependencies
- **Database driver**: `better-sqlite3` `^12.4.1` вЂ” already installed
- **Schema location**: `src/db/schema.ts` вЂ” Drizzle table definitions
- **Migrations**: Generated by `drizzle-kit generate` into `drizzle/migrations/`
- **Migration execution**: Use `drizzle-orm/better-sqlite3/migrator` to run migrations on app startup (push-based for MVP simplicity is also acceptable via `drizzle-kit push`)
- **Repository boundary**: Only `src/repositories/` may import the database client. Routes and services never touch DB directly.
- **Data flow**: Route в†’ Service в†’ Repository в†’ DB (this story establishes Repository в†’ DB)

### Technical Requirements

- **Database naming**: `snake_case` for all tables, columns, indexes
  - Table: `short_urls`
  - Columns: `id`, `short_code`, `original_url`, `normalized_url`, `created_at`
  - Index: `idx_short_urls_short_code`
- **API/domain naming**: `camelCase` for TypeScript interfaces and returned domain objects
- **File naming**: kebab-case (e.g., `short-url-repository.ts`, `schema.ts`, `client.ts`)
- **Type naming**: PascalCase (e.g., `ShortUrlRecord`, `ShortUrlInsert`)
- **Timestamps**: Store as ISO 8601 text strings in SQLite (SQLite has no native timestamp type). Use `new Date().toISOString()` at insert time.
- **Primary key**: Integer autoincrement вЂ” use Drizzle's `integer('id').primaryKey({ autoIncrement: true })`
- **Unique constraints**: `short_code` (unique + indexed), `normalized_url` (unique) вЂ” these enforce FR12 (short code uniqueness) and FR6 (duplicate URL detection)

### Drizzle Configuration

`drizzle.config.ts` at repo root:
```typescript
import { defineConfig } from 'drizzle-kit'

export default defineConfig({
  schema: './src/db/schema.ts',
  out: './drizzle/migrations',
  dialect: 'sqlite',
  dbCredentials: {
    url: process.env.DATABASE_PATH || './data/urls.db'
  }
})
```

### Database Plugin Pattern

Follow Fastify plugin conventions. The database plugin should:
1. Create better-sqlite3 connection from `app.config.databasePath`
2. Create parent directories for the SQLite file if they don't exist
3. Run Drizzle migrations (or use `drizzle-kit push` approach)
4. Decorate `app` with `db` (the Drizzle database instance)
5. Register an `onClose` hook to close the better-sqlite3 connection

- **Config pattern**: `loadConfig()` returns typed `AppConfig` with `databasePath` field вЂ” use this for DB connection

- **Type augmentation**: `src/types/fastify.d.ts` extends `FastifyInstance` with `config` вЂ” extend similarly with `db`
- **Existing deps**: `better-sqlite3`, `drizzle-orm`, `drizzle-kit` already installed in package.json
- **Test pattern**: Tests use Vitest in `tests/unit/` вЂ” follow the same pattern
- **Module system**: ESM with `.js` extensions in imports (NodeNext module resolution)

### Testing Standards

- **Framework**: Vitest `4.1.0`
- **Test DB**: Use a temp file (e.g., `:memory:` or `os.tmpdir()` path) for test isolation вЂ” do not use the app's configured DATABASE_PATH
- **Setup/teardown**: Create a fresh DB and run migrations before each test (or test suite). Close connection after.
- **Assertions**: Verify returned objects are typed `ShortUrlRecord` with correct camelCase fields, not raw snake_case DB rows
- **Edge cases**: Test unique constraint violations throw predictable errors
